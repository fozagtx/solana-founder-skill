# First Users and Acquisition for Solana Builders

Use this when the user asks how to get first users, first integrations, first paying customers, developer adoption, contributors, partners, or community traction.

## Core Principle

For early Solana projects, acquisition is usually founder-led and specific. The goal is not "growth." The goal is to reach the exact people who already feel the pain and ask them for a small concrete next step.

Pick one channel for the first sprint. Do not mix 5 channels before learning from one.

## Channel Selection

| Situation | Start With |
|-----------|------------|
| B2B protocol, infra, wallet, or devtool buyer | Ecosystem outbound |
| Developer tool, SDK, template, CLI | Developer dry-run plus GitHub/Discord issue strategy |
| Consumer app, game, creator, NFT, community product | Community problem-thread strategy |
| DeFi product needing liquidity or integrations | Partner/integration outbound |
| Hackathon or grant project | Reviewer narrative plus demo-user proof sprint |
| Open-source infrastructure | Docs/examples distribution plus contributor loop |
| Security, monitoring, analytics, support tooling | Teardown or diagnostic outreach |

## Channel 1: Ecosystem Outbound

Best for:

- Protocols
- Wallet teams
- Infra providers
- Devtool buyers
- DAO ops teams
- Game studios
- Data/API buyers

Process:

1. Define one ICP slice.
2. Build a list of 25 to 50 exact teams.
3. For each team, find one verifiable trigger: launch, incident, integration, new feature, job post, support issue, hackathon project, governance proposal, or public roadmap item.
4. Send 10 precise messages per day.
5. Ask for a small next step, not a full sales call.

Message template:

```text
Subject: quick idea for [team/project]

Hey [Name] - noticed [specific trigger].

We are helping [ICP] solve [pain] by [mechanism]. For [team], I think the highest-leverage use case is [one concrete observation].

Worth sending a 2-minute teardown, or not a priority right now?
```

Follow-up:

```text
Tiny follow-up on this. If [pain] is owned by someone else on the team, who should I send the teardown to?
```

## Channel 2: Developer Dry-Run Distribution

Best for SDKs, APIs, templates, CLIs, frameworks, and tools.

Playbook:

1. Create one "time to first value" task.
2. Write a quickstart that completes one meaningful Solana action.
3. Recruit 5 target developers from GitHub, Discord, hackathons, Stack Exchange, or your network.
4. Watch them try it.
5. Turn every failure into docs, examples, or product fixes.
6. Publish the improved quickstart and ask the same developers to retry or share.

DM template:

```text
Hey [Name] - saw you are building [project/context]. I am testing a Solana [SDK/template/tool] for [specific job].

Could I watch you try one task for 15 minutes? The goal is to find where the docs fail, not to pitch you.
```

Success metrics:

- Completion time
- Number of setup failures
- Number of unprompted questions
- Whether they use it in a real repo
- Whether they share it with another builder

## Channel 3: Problem-Thread Community Strategy

Best when pain is already discussed in public.

Where:

- Project Discord support channels
- Protocol forums
- Solana Stack Exchange
- GitHub issues
- X threads
- Telegram groups
- Hackathon channels

Rules:

- Lead with useful diagnosis.
- Do not spam links.
- Ask permission before moving to DMs.
- Capture the exact language people use.
- Turn repeated questions into docs, demos, and product pages.

Comment template:

```text
This usually breaks because [specific cause].

The quick check is:
1. [step]
2. [step]
3. [step]

I am testing a small tool/checklist for this exact issue. Happy to tailor it to your case if useful.
```

## Channel 4: Teardown or Diagnostic Outreach

Best for visible workflows:

- Wallet onboarding
- Landing page
- Docs
- Transaction UX
- Support flow
- DeFi position flow
- Developer setup
- Security posture snapshot

Offer:

```text
I recorded a 2-minute teardown with 3 fixes for [specific flow]. Want it?
```

If yes, send:

```text
Here are the 3 fixes:
1. [specific]
2. [specific]
3. [specific]

If useful, I can implement or validate this over [timeframe] for [price/commitment].
```

## Channel 5: Integration and Partner Distribution

Best when one integration unlocks credibility.

Targets:

- Wallets
- Protocols
- RPC/indexing providers
- Marketplaces
- Data platforms
- Hackathon organizers
- DAO tooling
- Creator platforms
- Game launchers

Partner pitch:

```text
[Partner] already serves [ICP]. We can add [specific user outcome] without asking your team to build it from scratch.

Proposed 2-week integration:
- [deliverable]
- [deliverable]
- [public artifact]

Win for you:
- [benefit to partner users]
- [support/revenue/retention/activation benefit]

Ask:
- [API access, grant, intro, co-marketing, integration review, or pilot fee]
```

## Channel 6: Open-Source Adoption Loop

Best when trust and developer adoption matter.

Loop:

1. Publish a narrow repo that solves one pain.
2. Include a 5-minute quickstart.
3. Add one production-like example.
4. Invite issue reports from exact ICPs.
5. Fix the top 3 blockers fast.
6. Publish a changelog with user language.
7. Offer paid help for teams adopting it in production.

Do not use open-source as an excuse to avoid sales. A repo with no user conversations is not distribution.

## Diagnose No Traction

After 50 targeted attempts or 7 days, inspect:

1. ICP mismatch
2. Trigger not urgent
3. Message too generic
4. Solana advantage unclear
5. Ask too large
6. Artifact not credible
7. Wrong channel
8. No follow-up

Fix the earliest issue first.
