# Proof Assets for Solana Products

Use this when the user provides interviews, pilot notes, testimonials, support threads, demo feedback, transaction evidence, or user stories and asks for case studies, proof points, social copy, or launch materials.

## Core Principle

Proof converts when it shows a painful before state, a credible trigger, the product in action, and a concrete result. Never invent metrics, quotes, team names, transaction volume, revenue, integrations, or user outcomes.

If a proof asset is thin, still produce a usable draft, but list the gaps clearly.

## Inputs

Accept:

- Interview transcript
- Pilot notes
- User support thread
- Discord/Telegram/X conversation
- Deal notes
- Grant milestone update
- Product analytics summary
- On-chain transaction signatures or dashboard screenshots
- Demo feedback

Use timestamps, line numbers, paragraph numbers, transaction signatures, issue links, or message IDs as citation anchors when available. If none exist, number paragraphs.

## Case Study Structure

Produce these sections:

### 1. Customer or User Snapshot

- Name or anonymized label
- Role or team type
- Solana category
- Why this user matches the ICP
- Citation

### 2. Before State

Describe:

- The workflow or transaction path that was broken
- The tools or manual process used
- The cost in time, risk, support load, failed transactions, missed integrations, churn, or user confusion
- The user's own language

Every claim needs a citation.

### 3. Trigger Moment

Name the event that made the pain urgent:

- Launch
- Incident
- Hackathon deadline
- New integration
- User support spike
- Failed transaction flow
- Grant milestone
- Audit preparation
- Partner request

### 4. Product Intervention

Explain what the product did in concrete terms:

- Setup
- Integration
- Workflow
- Dashboard
- Alert
- API
- SDK
- Template
- Analysis
- Support process

Avoid broad claims like "revolutionized" or "transformed" unless the evidence supports them.

### 5. Outcome

Format every metric as:

```text
Metric:
Before:
After:
Timeframe:
Evidence:
```

If no quantified metric exists, write:

```text
No quantified metric provided.
```

Then list what to ask next.

### 6. Pull Quotes

Choose up to 3 quotes:

- Under 30 words
- Verbatim or lightly trimmed with brackets
- Specific, emotional, or decision-relevant
- Cited

Themes:

- Pain
- Decision
- Outcome

### 7. Proof Bullets

Create 3 to 5 bullets usable on a grant, landing page, pitch, or launch post.

Example:

```text
- Reduced first setup time from 42 minutes to 11 minutes across 4 developer dry-runs.
- Identified 3 recurring wallet-support failure modes from 28 support tickets.
- Secured 2 design partners for the next pilot milestone.
```

### 8. Social or Launch Variant

Write a short post that leads with the proof, not hype.

```text
[ICP] were struggling with [pain].

We tested [product] with [evidence].

The signal: [result].

Next: [ask].
```

### 9. Gaps to Close

List follow-up questions:

- What was the baseline?
- What changed after using the product?
- Can we cite the team name?
- Can we show screenshots or transaction signatures?
- What would make this a weekly habit?
- What would they pay or commit to next?

## Grant Update Format

Use when writing milestone updates:

```text
Milestone:

What shipped:

Evidence:

Users reached:

What changed because of feedback:

Risks:

Next milestone:

Ask:
```

## Launch Proof Format

Use when writing public launch copy:

```text
Who it is for:

Problem:

What is live:

Proof:

How to try it:

What feedback is needed:

What comes next:
```

## Quality Checks

Before returning proof assets, verify:

1. Every specific claim has a citation or is clearly labeled as uncited.
2. No metric is invented.
3. No quote is fabricated or over-polished.
4. The before state is concrete.
5. The product intervention is understandable.
6. The asset includes a next ask.

If the asset sounds like generic marketing, rewrite it around specific user behavior.
