# Win/Loss Pattern Extraction for Solana Projects

Use this when the user provides a batch of closed-won and closed-lost pilots, user interviews, sales notes, grant feedback, churn notes, support tickets, demo feedback, or integration outcomes.

## Core Principle

One lost pilot is an anecdote. Ten lost pilots are a signal about ICP, promise, trust, timing, or channel. The job is to find patterns that change what the founder does next.

Read the full batch before writing. Count repeated signals. Separate buyer/user language from founder interpretation.

## Inputs

The user may provide:

- CRM notes
- Outreach replies
- Interview transcripts
- Demo feedback
- Support tickets
- Discord/Telegram messages
- GitHub issues
- Grant or hackathon reviewer comments
- Churn notes
- Pilot outcomes

If won/lost status is unclear, ask once. If still unclear, classify by behavior:

- Won: paid, integrated, used repeatedly, referred, gave data access, committed next step
- Lost: no-show, no decision, explicit no, churned, refused integration, no follow-up, rejected grant

## Report Structure

Start with a one-paragraph executive summary naming the 2 to 3 biggest patterns.

Then produce these sections.

### 1. Loss Reason Patterns

For each top loss reason:

```text
Stated reason:
Likely real reason:
Gap:
Frequency:
Representative quote:
Evidence:
Fix:
```

Solana loss patterns to watch:

- Solana necessity unclear
- Security/trust not handled
- Integration effort too high
- Buyer lacks ownership of the problem
- Grant/demo interest did not convert
- Token optics created resistance
- Category was too vague
- Product was a feature, not a workflow
- Developer setup failed before first value
- Status quo scripts/spreadsheets were good enough

### 2. Win Reason Patterns

For each win reason:

```text
Win reason:
What actually tipped the decision:
Frequency:
Representative quote:
Evidence:
How to repeat:
```

Look for:

- Trigger event
- Urgent deadline
- Team pain
- Integration partner
- Clear proof artifact
- Fast setup
- Trusted intro
- Strong Solana-native advantage

### 3. Competitor and Status Quo Clusters

Treat "do nothing" as a competitor.

Cluster by:

- Internal build
- Existing vendor
- Manual ops
- Open-source script
- EVM-first tool
- Generic SaaS
- Another Solana protocol/tool
- No decision

For each:

```text
Cluster:
Count:
Why it won:
Where it is vulnerable:
ICP segment affected:
Messaging implication:
```

### 4. ICP Segment Performance

Slice by whatever data exists:

- Protocol category
- Team size
- Buyer role
- Developer maturity
- Chain/ecosystem exposure
- Mainnet vs devnet stage
- Hackathon vs funded startup
- Wallet, DeFi, infra, game, DAO, consumer, creator, compliance
- Geography or regulatory sensitivity if mentioned
- Trigger event

Be honest about sample size.

### 5. Killer Objections

List objections that ended deals or adoption:

```text
Objection:
Frequency:
Where it appeared:
Representative quote:
Needed proof or product change:
```

### 6. Decision-Moment Language

Pull direct phrases from users:

- Language at yes
- Language at no
- Language at no decision
- Language at "maybe later"

Do not paraphrase. If exact quotes are unavailable, label them as summarized.

### 7. Recommended Messaging Shifts

Give 3 shifts:

```text
Pattern:
Stop saying:
Start saying:
Where to deploy:
Evidence needed:
```

Weak:

```text
Emphasize security.
```

Strong:

```text
Stop saying "production-ready analytics." Start saying "read-only failure-mode analysis for wallet support teams, no signing authority required." Deploy in first outbound message and security FAQ.
```

### 8. Product and Roadmap Implications

Name what to change:

- ICP
- promise
- onboarding
- docs
- demo path
- integration scope
- pricing
- proof
- security posture
- acquisition channel

Do not recommend a feature just because one loud prospect asked for it. Tie recommendations to repeated evidence.


## Quality Checks

Before returning:

1. Count frequencies where possible.
2. Separate stated reasons from likely real reasons.
3. Quote users or label summaries honestly.
4. Identify status quo as a competitor.
5. Recommend changes tied to repeated patterns.
6. Name the data gaps that weaken confidence.
